18 * Overview of Error.php
19 *
20 * This file defines the Error class in the orange\framework namespace.
21 * It is a centralized error handler for the framework, designed to capture exceptions,
22 * render meaningful error responses, and send them back to the client in a format appropriate to
23 * the request type (HTML, JSON/AJAX, CLI). It extends Singleton, ensuring a single instance is used application-wide,
24 * and uses the ConfigurationTrait for flexible setup.
25 *
26 * ⸻
27 *
28 * 1. Core Purpose
29 *  •   Capture exceptions or error codes.
30 *  •   Load error details into a data service.
31 *  •   Select an appropriate error view (based on environment, request type, or error code).
32 *  •   Render the view or provide a raw fallback output.
33 *  •   Send the response to the client with correct HTTP code, content type, and exit code.
34 *
35 * ⸻
36 *
37 * 2. Key Properties
38 *  •   Service instances:
39 *  •   $data → holds error data (message, code, trace, etc.).
40 *  •   $input → provides request type information (HTML, AJAX, CLI).
41 *  •   $view → used to render error templates.
42 *  •   $output → sends headers, status codes, and content to the client.
43 *  •   $container → dependency injection container for resolving services.
44 *  •   Error context:
45 *  •   $code (default 500) → application error code.
46 *  •   $httpCode → optional HTTP status code from the exception.
47 *  •   $requestType → request channel (cli, html, ajax).
48 *  •   $errorViewDirectory, $envDirectory, $requestTypeDirectory → directories to search for error views.
49 *  •   $viewFile, $outputContent → resolved error template file or rendered output.
50 *
51 * ⸻
52 *
53 * 3. Constructor Behavior
54 *
55 * When instantiated, the class:
56 *  1.  Loads configuration and resolves dependencies (data, input, view, output).
57 *  2.  Determines the environment (production, development, etc.) and request type.
58 *  3.  If given a Throwable, extracts details (message, code, file, line, stack trace).
59 *  4.  Supports custom exception methods (getHttpCode, getOutput, decorate) for enhanced control.
60 *  5.  Attempts to resolve an error view file based on error/HTTP code.
61 *  6.  If no template is found, falls back to raw output formatting (plain text, JSON, or HTML <pre>).
62 *  7.  Immediately sends output to the client and terminates execution.
63 *
64 * ⸻
65 *
66 * 4. Key Methods
67 *  •   show($code, $message, $options) → manually trigger and render an error with code and message.
68 *  •   sendResponseCode() → sends appropriate HTTP status (defaults to 500).
69 *  •   sendMimeType() → sets response type (html, json, or defaults).
70 *  •   sendOutput($content, $exitCode) → flushes output buffer, writes content, sends headers, and exits.
71 *  •   renderViewBasedOnCode($code, $httpCode) → looks for an error template by code or status.
72 *  •   findView($view) → searches directories in a defined order (env + request type fallbacks).
73 *  •   viewRaw() → fallback plain output if no template found (formats based on request type).
74 *  •   getService($name, $arguments) → resolves dependencies from the container, falling back to Orange defaults.
75 *
76 * ⸻
77 *
78 * 5. Error Rendering Logic
79 *  1.  Preferred → environment + request type-specific error view (e.g., errors/dev/html/404.php).
80 *  2.  Fallbacks → environment-specific or general error views.
81 *  3.  Last resort → raw inline response (JSON, HTML <pre>, or CLI print).
82 *
83 * ⸻
84 *
85 * 6. Big Picture
86 *
87 * Error.php is the last line of defense in the framework.
88 * It ensures that all errors and exceptions result in a consistent,
