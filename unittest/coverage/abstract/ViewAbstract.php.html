23/**
24 * Overview of ViewAbstract.php
25 *
26 * This file defines the ViewAbstract class in the orange\framework\abstract namespace.
27 * It is the base class for all view rendering engines in the Orange framework.
28 * Concrete engines (like PHP views, Twig, Markdown, etc.)
29 * extend it to provide specific rendering behavior, while sharing the same core setup and utilities.
30 * It also enforces the Singleton pattern and implements the ViewInterface.
31 *
32 * ⸻
33 *
34 * 1. Core Purpose
35 *  •   Provide a foundation for rendering views (templates) in different formats.
36 *  •   Manage configuration, view paths, aliases, caching, and data injection.
37 *  •   Allow support for dynamic view resolution based on routing.
38 *  •   Handle file-based and string-based rendering consistently.
39 *
40 * ⸻
41 *
42 * 2. Key Properties
43 *  •   $search → a DirectorySearch instance for locating view files.
44 *  •   $data → a data source (implements DataInterface) passed into views.
45 *  •   $router → optional router instance for resolving dynamic view names.
46 *  •   $debug → toggles debug mode (forces re-rendering, no caching).
47 *  •   $allowDynamicViews → enables placeholders like $c/$m in view paths.
48 *  •   $tempDirectory → directory for temporary cached view files (string templates).
49 *  •   $alias → maps view names to alternate paths.
50 *  •   $subPathSize → determines subdirectory depth for hashing string templates.
51 *  •   $changeableTypeCheck → defines which properties can be updated at runtime and their type checks.
52 *
53 * ⸻
54 *
55 * 3. Initialization
56 *  •   Constructor merges config, sets debug/dynamic flags, validates the temp directory, loads aliases, and builds the DirectorySearch utility with configured view paths.
57 *  •   Can also preload resources into the search utility.
58 *
59 * ⸻
60 *
61 * 4. Key Methods
62 *  1.  Searching and Aliasing
63 *  •   search() → returns the directory search utility.
64 *  •   addAlias($view, $aliasView) → registers a view alias.
65 *  •   resolveAlias($view) → replaces a view name with its alias.
66 *  2.  Rendering Views
67 *  •   render($view, $data, $options) → locates a view file and renders it with merged data.
68 *  •   renderString($string, $data, $options) → renders directly from a string (compiled into a temp file).
69 *  •   generate($__viewFilePath, $__dataArray) → internal method that executes the view file with provided data.
70 *  3.  Dynamic View Resolution
71 *  •   resolveDynamicView($view) → replaces placeholders like $c (controller), $m (method), $1, $2 (namespace segments) using the matched route.
72 *  4.  Data Handling
73 *  •   data($data) → merges new data with the view’s internal DataInterface.
74 *  5.  Configuration
75 *  •   change($name, $value) → safely update configurable properties (e.g., enable debug mode, change temp directory).
76 *  6.  File Safety
77 *  •   isFileWritable($file) → ensures the target file or directory is writable (creates directories if needed).
78 *
79 * ⸻
80 *
81 * 5. Error Handling
82 *  •   Throws ViewNotFound if a view file cannot be located.
83 *  •   Throws DirectoryNotWritable or FileNotWritable if caching directories are inaccessible.
84 *  •   Throws InvalidValue for incorrect config changes.
85 *  •   Wraps low-level errors into framework exceptions for consistency.
86 *
87 * ⸻
88 *
89 * 6. Big Picture
90 *  •   ViewAbstract is the backbone of Orange’s view system.
91 *  •   It standardizes how views are located, resolved, cached, and rendered, while allowing flexible engines to be built on top.
92 *  •   By combining search, aliasing, dynamic resolution, and rendering, it ensures all view engines behave consistently across the framework.
93 *
94 * ⸻
95 *
96 * Recommendation: Treat ViewAbstract as the template engine foundation.
97 * All custom view renderers should extend it to inherit search, caching, and rendering logic.
